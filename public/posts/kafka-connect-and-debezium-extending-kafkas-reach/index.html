In previous parts of this series, we explored Kafka’s core architecture and how Kafka Streams enables real-time processing. Now, it’s time to look at Kafka Connect and Debezium—two essential tools that extend Kafka’s reach by integrating it with other systems and enabling real-time change data capture (CDC).

If Kafka is the powerful coffee shop, Kafka Connect and Debezium are like an advanced ordering system that pulls orders from various locations (databases, file systems, etc.) and streams them seamlessly into the shop’s workflow, ensuring real-time updates.

1. What is Kafka Connect?#

Kafka Connect is a scalable and flexible framework that allows you to stream data in and out of Kafka topics using pre-built connectors. These connectors make it easy to integrate Kafka with external systems, such as databases, file systems, or cloud services.

There are two types of connectors in Kafka Connect:

  1. Source Connectors: These pull data from external systems into Kafka topics.
  2. Sink Connectors: These push data from Kafka topics into external systems.

Kafka Connect is designed to be:

  • Distributed and Scalable: It can scale across multiple nodes to handle large amounts of data.
  • Fault-Tolerant: It ensures data is not lost if a connector fails by using offsets to track the position of the data.
  • Easily Configurable: With pre-built connectors, it’s easy to set up and manage.

2. How Kafka Connect Works#

Kafka Connect enables data pipelines between external systems and Kafka. For example, you could use a source connector to pull data from a MySQL database and stream it into a Kafka topic. Then, you could use a sink connector to push the processed data from Kafka into a data warehouse like Amazon Redshift or an analytics platform like Elasticsearch.

  • Source Connectors: Continuously read data from a source system (like a database) and publish it to Kafka topics.
  • Sink Connectors: Read data from Kafka topics and push it to a target system (like a file system, NoSQL database, or cloud storage).

Example: Setting Up a MySQL Source Connector#

Let’s say you want to sync data from a MySQL database into Kafka in real time. You can use Kafka Connect with a MySQL source connector:

  1. Install the MySQL Connector: Download and install the MySQL Kafka connector as a plugin for Kafka Connect.

  2. Configure the Connector: The connector is configured via a simple JSON file. Below is an example of a MySQL source connector configuration:

3. Debezium: Real-Time Change Data Capture (CDC)#

One of the most powerful use cases of Kafka Connect is Change Data Capture (CDC), which enables you to track real-time changes in a database and stream them into Kafka topics. This is where Debezium comes into play.

Debezium is an open-source platform that sits on top of Kafka Connect and provides CDC capabilities for databases like MySQL, PostgreSQL, SQL Server, MongoDB, and more. It captures every insert, update, and delete operation and streams those changes into Kafka.

How Debezium Works#

Debezium monitors the transaction logs (also called binlogs or redo logs) of a database, capturing every change made to the data. These changes are then streamed to Kafka topics in real time.

  • Source Connector: Debezium runs as a Kafka Connect source connector, continuously pulling changes from the database.
  • Kafka Topics: Each change is published to a specific Kafka topic, typically with both the before and after states of the data.
  • Schema Registry: Debezium integrates with Kafka’s Schema Registry to manage data schemas, ensuring compatibility between different systems.

4. Use Cases for Kafka Connect and Debezium#

4.1 Database Replication#

With Kafka Connect and Debezium, you can replicate data from one database to another in real time. For example, you could replicate data from a MySQL database running in a central office to a PostgreSQL database in a remote branch or vessel.

  • How It Works:
    • Use Debezium’s MySQL connector to capture changes in the central office database.
    • These changes are streamed to Kafka topics.
    • A sink connector reads from the Kafka topic and applies the changes to the PostgreSQL database in real time.

This ensures that both databases are kept in sync with minimal delay.

4.2 Event-Driven Architectures#

In a microservices architecture, Debezium can be used to trigger events based on changes in a database. For example, if an order is updated in a database, Debezium captures the change and streams it to Kafka. This can trigger downstream services to send notifications or update inventory systems.

  • How It Works:
    • When a customer places an order, the order service writes it to the database.
    • Debezium captures the order and streams the event to Kafka.
    • Other microservices consume this event and act accordingly, such as sending a confirmation email or updating the inventory.

4.3 Real-Time Analytics#

Kafka Connect and Debezium can also be used for real-time analytics by streaming data from transactional databases to analytics platforms like Elasticsearch, Redshift, or data lakes.

  • How It Works:
    • Debezium captures changes in the transactional database.
    • Kafka Connect streams these changes to an analytics platform.
    • Real-time dashboards can be built on top of this data to provide instant insights.

5. Bidirectional Data Synchronization Using Debezium#

One of the advanced use cases for Debezium is bidirectional data synchronization. This is useful when you need to synchronize data between two or more systems (such as a central office database and a vessel database) in real time.

Here’s how it works:

  1. Debezium at the Source: Debezium captures changes from the source database (e.g., central office) and streams them into Kafka with a custom source header identifying where the change originated.

  2. Debezium at the Destination: Another instance of Debezium captures changes from the destination database (e.g., vessel) and streams them to Kafka, also tagging the changes with a source header.

  3. Conflict Resolution: By using the source header, you can configure the consumers at both ends (office and vessel) to apply changes from the opposite location, avoiding circular replication and ensuring that no changes are replayed back to the source.

This setup ensures conflict-free bidirectional data synchronization between two databases, even if they are geographically separated.

6. Example: Syncing SQL Server Databases in Real-Time#

Suppose you need to keep a SQL Server database in a central office in sync with another SQL Server database on a vessel. Here’s how you can set it up using Kafka Connect and Debezium:

  1. Debezium in the Office:

    • Install Debezium’s SQL Server connector.
    • Capture all changes (inserts, updates, deletes) from the central office SQL Server and stream them into Kafka topics.

  2. Debezium on the Vessel:

    • Install Debezium’s SQL Server connector on the vessel database.
    • Stream changes from the vessel database into Kafka topics and tag them with a source header as “vessel.”

  3. Bidirectional Sync:

    • Configure consumers in the office to apply changes from the vessel (and vice versa).
    • Ensure that only changes originating from the opposite source are applied, preventing a bidirectional loop.

7. Example: Setting Up Debezium with SQL Server#

Let’s walk through how to set up a Debezium SQL Server connector to capture changes in a SQL Server database and stream them into Kafka topics.

Step 1: Install Kafka Connect and Debezium SQL Server Connector#

You need to have Kafka Connect running with the Debezium SQL Server connector installed.

  1. Download Debezium SQL Server Connector: Download and extract the SQL Server connector for Kafka Connect. You can find it on Debezium’s official site or by using the Maven repository.

    1

    curl -O https://repo1.maven.org/maven2/io/debezium/debezium-connector-sqlserver/1.9.5.Final/debezium-connector-sqlserver-1.9.5.Final-plugin.tar.gz

  2. Extract the Connector: Extract the connector and place it in the plugin.path directory of your Kafka Connect installation.

    1

    tar -xvf debezium-connector-sqlserver-1.9.5.Final-plugin.tar.gz -C /path/to/kafka-connect/plugins

Step 2: Configure SQL Server for CDC#

Ensure Change Data Capture (CDC) is enabled on your SQL Server database. CDC needs to be enabled for the specific tables you want to monitor for changes.

  1. Enable CDC at the Database Level:

    1
2

    USE master;
EXEC sys.sp_cdc_enable_db;

  2. Enable CDC on a Specific Table:

    Suppose you want to capture changes from the orders table. You’ll need to enable CDC for this table.

    1
2
3
4
5

    USE your_database;
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name = N'orders',
    @role_name = NULL;

    Once enabled, SQL Server will start tracking changes (inserts, updates, deletes) in this table.

Step 3: Configure Debezium SQL Server Connector#

You now need to configure the Debezium SQL Server connector to capture changes and send them to Kafka topics. Create a JSON configuration file for your SQL Server connector.

{
  "name": "sqlserver-source-connector",
  "config": {
    "connector.class": "io.debezium.connector.sqlserver.SqlServerConnector",
    "database.hostname": "sqlserver-host",
    "database.port": "1433",
    "database.user": "your_user",
    "database.password": "your_password",
    "database.dbname": "your_database",
    "database.server.name": "sqlserver",
    "database.history.kafka.bootstrap.servers": "localhost:9092",
    "database.history.kafka.topic": "schema-changes.sqlserver",
    "table.include.list": "dbo.orders",
    "database.encrypt": "false",  // Set true if you use encrypted communication
    "snapshot.mode": "initial",  // Captures initial data snapshot
    "poll.interval.ms": "1000"
  }
}

Key parameters:

  • database.hostname: Hostname of your SQL Server instance.
  • database.user: Username to connect to the SQL Server database.
  • database.password: Password for the SQL Server user.
  • database.dbname: The database name you want to capture changes from.
  • table.include.list: The specific table(s) for which you want to capture changes. Here, dbo.orders is included.
  • snapshot.mode: When set to “initial,” the connector will first capture the current state of the table, followed by capturing incremental changes.

Step 4: Start Kafka Connect with the SQL Server Connector#

Now that the connector configuration is ready, start Kafka Connect in distributed mode and load the connector configuration.

You can submit the connector configuration via Kafka Connect’s REST API:

1
2
3

curl -X POST -H "Content-Type: application/json" \
    --data @sqlserver-connector-config.json \
    http://localhost:8083/connectors

This will start capturing changes from the SQL Server database and streaming them into Kafka topics.

Step 6: Bidirectional Synchronization (Optional)#

If you are setting up bidirectional synchronization between two SQL Server databases (e.g., an office database and a vessel database), you can run two instances of the Debezium connector—one for each database.

You can include a custom source header to differentiate changes originating from the office and vessel databases. Consumers can use this header to avoid replaying the same changes back to the original database.

For example, you might tag changes from the office with a "source": "office" header and changes from the vessel with "source": "vessel". This prevents circular replication loops.

Conclusion: Building Real-Time Data Pipelines with Kafka Connect and Debezium#

Kafka Connect and Debezium extend Kafka’s power by allowing you to integrate with external systems and capture changes from databases in real-time. Whether you’re building a real-time data pipeline, syncing databases across multiple locations, or enabling event-driven microservices, Kafka Connect and Debezium provide the tools you need to stream data seamlessly and efficiently.
